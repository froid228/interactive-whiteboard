# Fuzzing Report

## Цель проверки

Фаззинг-тестирование используется для проверки устойчивости REST API и Socket.IO-канала к некорректным, пустым, граничным и случайным входным данным. Основная цель проверки: убедиться, что приложение возвращает контролируемые ошибки, а не необработанные серверные ошибки `500`, падение процесса или зависание Socket.IO acknowledgement.

## Инструмент

В проекте используются два собственных сценария:

- `server/scripts/fuzz-api.js` — негативные, граничные и рандомизированные HTTP-запросы к REST API;
- `server/scripts/fuzz-socket.js` — негативные, граничные и рандомизированные Socket.IO-события к realtime-каналу доски.

Команды запуска:

```bash
cd server
npm run fuzz
npm run fuzz:socket
```

Для последовательного запуска REST и Socket.IO fuzzing:

```bash
cd server
npm run fuzz:all
```

Для запуска против нестандартного адреса API:

```bash
FUZZ_API_URL=http://localhost:5001/api npm run fuzz
```

Для запуска Socket.IO fuzzing против нестандартного backend:

```bash
FUZZ_SOCKET_URL=http://localhost:5001 FUZZ_API_URL=http://localhost:5001/api npm run fuzz:socket
```

Количество случайных payload-проверок можно изменить переменной окружения:

```bash
RANDOM_FUZZ_CASES=200 npm run fuzz
RANDOM_SOCKET_CASES=200 npm run fuzz:socket
```

## Проверяемые группы сценариев

### REST API

- регистрация с пустым телом запроса;
- регистрация с коротким именем;
- регистрация с некорректным email;
- регистрация с коротким паролем;
- повторная регистрация с уже существующим email;
- вход с пустым телом запроса;
- вход с неверным паролем;
- обращение к защищенному `/auth/me` без токена;
- обращение с невалидным Bearer-токеном;
- получение списка досок без авторизации;
- создание доски с пустым, коротким и слишком длинным названием;
- создание и обновление досок с некорректными полями;
- попытки получить, изменить и удалить чужую доску;
- выдача доступа с некорректными email;
- повторная выдача доступа;
- удаление несуществующего участника;
- проверка журнала активности досок;
- рандомизированные payload-проверки для регистрации, входа, создания досок, обращения к невалидным id и отправки сообщений.

### Socket.IO

- handshake без токена;
- handshake с некорректным JWT;
- успешное подключение с валидным JWT;
- `join-board` с валидным id, строковым id, объектом вместо id и доступом чужого пользователя;
- `draw-segment` с некорректным `boardId`, пустым сегментом, некорректным цветом и превышением лимита точек;
- `board-snapshot` с неверным типом snapshot и превышением лимита элементов;
- `clear-board` с некорректным id доски;
- `send-message` с пустым сообщением, слишком длинным сообщением и попыткой отправки без доступа к доске;
- рандомизированные payload-проверки для событий `join-board`, `draw-segment`, `board-snapshot`, `clear-board`, `send-message`.

## Ожидаемый результат

Успешным считается результат, при котором каждая REST-проверка возвращает ожидаемый HTTP-статус, а каждая Socket.IO-проверка получает контролируемый acknowledgement с `ok: false`, ожидаемый отказ авторизации или безопасный timeout без падения процесса. Для некорректных REST-данных ожидаются клиентские ошибки `4xx`. Ответы `5xx`, падение backend-процесса и необработанные Socket.IO-ошибки считаются дефектом. В конце запуска скрипты выводят статистику `PASS`, `WARN` и `ERROR`.

## Итог для отчета

Фаззинг-сценарии добавлены в проект и запускаются командами `npm run fuzz`, `npm run fuzz:socket` или общей командой `npm run fuzz:all`. Они покрывают основные негативные сценарии авторизации, управления досками, выдачи доступа, проверки защищенных REST-маршрутов, Socket.IO handshake, realtime-события доски и рандомизированную подачу некорректных payload. Результаты запуска со статистикой `PASS/WARN/ERROR` можно приложить к отчету как скриншот терминала.
