# Fuzzing API Report

## Цель проверки

Фаззинг-тестирование используется для проверки устойчивости REST API к некорректным, пустым, граничным и случайным входным данным. Основная цель проверки: убедиться, что API возвращает контролируемые ответы `400`, `401`, `403`, `404` или `409`, а не необработанные серверные ошибки `500`.

## Инструмент

В проекте используется собственный сценарий `server/scripts/fuzz-api.js`, который выполняет набор негативных и граничных HTTP-запросов к запущенному backend, а также отдельный блок рандомизированных payload-проверок.

Команда запуска:

```bash
cd server
npm run fuzz
```

Для запуска против нестандартного адреса API:

```bash
FUZZ_API_URL=http://localhost:5001/api npm run fuzz
```

Количество случайных payload-проверок можно изменить переменной окружения:

```bash
RANDOM_FUZZ_CASES=200 npm run fuzz
```

## Проверяемые группы сценариев

- регистрация с пустым телом запроса;
- регистрация с коротким именем;
- регистрация с некорректным email;
- регистрация с коротким паролем;
- повторная регистрация с уже существующим email;
- вход с пустым телом запроса;
- вход с неверным паролем;
- обращение к защищенному `/auth/me` без токена;
- обращение с невалидным Bearer-токеном;
- получение списка досок без авторизации;
- создание доски с пустым, коротким и слишком длинным названием;
- создание и обновление досок с некорректными полями;
- попытки получить, изменить и удалить чужую доску;
- выдача доступа с некорректными email;
- повторная выдача доступа;
- удаление несуществующего участника;
- проверка журнала активности досок.
- рандомизированные payload-проверки для регистрации, входа, создания досок, обращения к невалидным id и отправки сообщений.

## Ожидаемый результат

Успешным считается результат, при котором каждая проверка возвращает ожидаемый HTTP-статус. Для некорректных данных ожидаются клиентские ошибки `4xx`. Ответы `5xx` считаются дефектом, так как означают необработанную серверную ошибку. В конце запуска скрипт выводит статистику `PASS`, `WARN` и `ERROR`.

## Итог для отчета

Фаззинг-сценарий добавлен в проект и запускается командой `npm run fuzz`. Он покрывает основные негативные сценарии авторизации, управления досками, выдачи доступа, проверки защищенных маршрутов и рандомизированную подачу некорректных payload. Результаты запуска со статистикой `PASS/WARN/ERROR` можно приложить к отчету как скриншот терминала.
